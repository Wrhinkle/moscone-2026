## The Highest Loop (swyx, AI Engineer)

[0:01:08](https://www.youtube.com/watch?v=htM02KMNZnk&t=68s)

swyx opened the conference with a short framing talk on loops as the core AI engineer skill: knowing which loop you are working in, going up a level for productivity and scale, going down a level for reliability and debugging. ## On AI and Knowledge (Pablo Castro, Microsoft)

[0:11:12](https://www.youtube.com/watch?v=htM02KMNZnk&t=672s)

Castro, a CVP and distinguished engineer at Microsoft, split knowledge into three categories: intrinsic (parametric, from training), extrinsic (retrieved and grounded), and learned (captured from doing the work). On intrinsic knowledge he traced a compressing timeline: IntelliSense in 1996, ML-ranked completions 22 years later, GitHub Copilot three years after that, then Opus 4.5 in late 2025 and OpenClaw early this year, "not a single line of code written by hand." He noted that Claude became generally available in Microsoft Foundry the day before the talk. On extrinsic knowledge he described Microsoft IQ as a set of entry points into ambient company data: Work IQ (SharePoint, email, calendar, chat), Fabric IQ (warehouses, Power BI), Foundry IQ (bring-your-own grounding data), and Web IQ. He argued evaluations repeatedly show combined retrieval methods beat any single method, and demoed Foundry knowledge bases: pick a model, set an effort level trading latency for quality, ground on blob storage, Parquet tables, and the web, and each knowledge base is exposed as an MCP server so existing harnesses can connect without glue code. Experts can drop down to Azure AI Search to control quantization, indexing algorithms, and chunking. He also flagged token efficiency as a design goal, aiming for the most information-dense answer in the fewest tokens. For learned knowledge he showed the Foundry agent optimizer in VS Code: generate a task-adherence eval from traces, run a roughly 45-minute optimization that hill-climbs the metric with a GEPA-style loop, then apply the winning configuration, producing instructions that emerged from reflection on real agent traces rather than handwriting.

## Rise of the Software Factory (Tereza Tížková, Factory)

[2:10:18](https://www.youtube.com/watch?v=htM02KMNZnk&t=7818s)

Tížková defined the software factory as the whole software lifecycle run with autonomy: collecting signals, prioritizing, orchestrating, executing, validating, and iterating, deployed in production for enterprises like EY and Adobe. It is not a coding agent, and not even a swarm of coding agents, since writing code is the easy part. Her three principles: agnostic, autonomous, always improving. Agnostic means working across Slack, GitHub, and existing subscriptions, and being model-agnostic; she cited Coinbase's cost curve, where token usage kept growing while spend flattened via default-model policies, caching, and routing. Factory's automatic model routing classifies task difficulty and picks the cheapest model predicted to clear the threshold, escalating on failure; their conservative benchmark shows about 25% savings plus reliability and failover benefits. Autonomous means Factory "missions": long-running sessions, some running for weeks, one customer mission ran 16 hours with validation taking about 40% of the process. The structure is an orchestrator that writes a validation contract before any code, worker agents that run in sequence (fresh context per handoff, each with parallel sub-agents), and two validator types: a scrutiny validator doing rigorous code checks and a user-testing validator that clicks through the running product in a virtual computer. Always improving covers a deferred context engine that progressively discloses tools (enterprises average hundreds of tools; hiding full schemas until needed saves 50% or more of tokens at scale), an agent-readiness framework for codebase hygiene, since unstructured codebases plus AI compound into worse code, and plugins plus auto-updating documentation. Her ending: humans decide what to build, agents decide how.

## Orchestras, not Factories (Charlie Holtz, Conductor)

[2:40:40](https://www.youtube.com/watch?v=htM02KMNZnk&t=9640s)

Holtz, co-founder of Conductor, a desktop app for managing a team of coding agents in one interface, distilled principles from watching top builders use the product. One: stay near the frontier, trying tools the day they ship; Conductor itself emerged from his team's Claude Code power use, cloning their repo five times before discovering worktrees. Two: don't try to beat the market. If a workflow would work for everyone, the labs will build it into the default harness; only invest in workflow when you have real alpha, meaning knowledge of your users or codebase the models lack. Three: create slop-free zones, parts of the codebase requiring strict human review. Conductor's migrations file needs human sign-off in CI, anything written in Slack is assumed human, and the team pours unusual effort into CLAUDE.md and skill files, the words whispered into the agent's ear at every session start. Four: feed the beast. Their internal agent (the CIA) writes every Slack message, Discord bug report, and meeting recording into Postgres and exposes a SQL tool. Five: free-range agents. He demoed the new Conductor version shipping that week, moving every workspace from a git worktree to a cloud sandbox that survives a closed laptop, with real-time visibility into teammates' workspaces and live co-editing, plus an API that let his OpenClaw agent ("Lord Crandon") spawn Conductor workspaces from Telegram. Sixth, the title: he dislikes the factory metaphor and wants tools that keep a human in the flow, conducting intermingled humans and agents rather than managing a production line.

## What we learned by analyzing 1M AI-generated PRs (Daksh Gupta, Greptile)

[3:05:01](https://www.youtube.com/watch?v=htM02KMNZnk&t=11101s)

Greptile reviews and tests pull requests for tens of thousands of teams, over a million PRs a month, including at Nvidia, Coinbase, and Scale. Gupta mined that data to test whether fully agent-written PRs hold up in commercial settings. Detection was the first problem: only about 1 in 100 PRs attribute an agent in the GitHub author field, so he combined co-authored-by trailers and agent-generated branch prefixes, finding that in April more than a quarter of reviewed PRs showed evidence of being largely agent-written, up from under 1% a year earlier. On quality, the signals were surprisingly even. Revert rates: about 1 per 1,000 for Codex, 3.5 for Devin, 2.5 for human-owned PRs, stable across PR sizes. Greptile-flagged issue rates (P0/P1/P2) were similar, with humans slightly more likely to introduce P0 errors than Devin, Codex, or Claude. Mean review rounds to mergeable: Devin just over 2, humans 2.2, Codex about 2.5. Failure patterns differed qualitatively on two months of recent data across millions of PRs: Claude was 1.5x more likely than people to produce a SQL injection error, Devin produced auth bypasses at half the rate of others, and N+1 query errors were far more common from Cursor. He also noted the median Greptile-using coder writes fewer than 50 commits a month while the 99th percentile approaches 1,000, making manual review infeasible at the high end. His conclusion: code validation should answer three questions (does the change violate the user contract, does it raise the future propensity of a violation, does it fulfill author intent), which Greptile approaches with codebase-aware review, blast-radius analysis, and sandboxed browser agents that try to break the change.

## Self-Improving Software Factories: The New Open Source Model (Zach Lloyd, Warp)

[4:55:21](https://www.youtube.com/watch?v=htM02KMNZnk&t=17721s)

Lloyd, Warp's founder and a former Google Docs principal engineer, said he ships frequently but has not written a line of code in six months. His thesis: software engineering becomes factory engineering, and every sizable project will run a loop of triage, spec, implement, review, verify, ship, and monitor, with humans at defined checkpoints. Warp open-sourced a couple of months earlier, partly to build a public factory: build.warp.dev shows issues flowing through agents and contributors, backed by over 60,000 GitHub stars, a couple hundred contributors, and over 800,000 active developers. On open source economics: when software is nearly free to build it is trivial to clone, so a great product alone is not a business; you need distribution, ecosystem, brand, or a data moat, and building in the open helps startups get there, with agent automations now managing the traditional pain of noisy issues and sloppy PRs. His factory tour: inputs arrive from trackers, Slack, terminals, and monitoring; a triage agent implements easy unambiguous issues directly; harder ones get a product spec and a tech spec from a spec agent; implementation runs in cloud sandboxes; review runs agent-first with humans applied as risk management; verification includes computer use producing videos and screenshots; monitoring feeds back to the top. He emphasized self-improvement loops, for example observer agents watching how a code-review agent's comments get corrected by senior engineers and updating its skills for the next run. In Q&A he said most companies should buy rather than build the infrastructure and focus on tuning the factory for their domain, that Warp is hiring more than ever, and that the joy trade-off is real: engineers will write less code and ship more.

## We're the Bottleneck, but We Don't Have to Be (Ido Salomon)

[5:26:49](https://www.youtube.com/watch?v=htM02KMNZnk&t=19609s)

Salomon, creator of AgentCraft and of MCP-UI (which became MCP Apps) and an MCP steering committee member, argued that humans are the constraint in agent orchestration: every spun-up agent demands steering, direction, and review, and at scale it is exhausting. His answer borrows from real-time strategy games. AgentCraft is a game-styled orchestrator where each unit on a map is a real agent (Claude Code, Codex, Open Code, OpenClaw), buildings represent functions like plugin management, terminal, and git, and the file system can be projected onto the map so you can see which agent is touching which directory, with heat maps of activity. Review load is handled with a review kit: full diffs, per-file views, and visual evidence including videos and screenshots of what changed, plus running several implementations in parallel and picking the best. For collaboration he demoed shared rooms where teammates (his wife, a product designer, in the demo) join, work is handed off or forked, and people and agents share a notice board, with mobile and Telegram access. The feedback that struck him most was accessibility: kids orchestrating agents, and a Starcraft player becoming productive. [5:49:09](https://www.youtube.com/watch?v=htM02KMNZnk&t=20949s)

## Fighting Slop with Slop (Vaibhav Gupta, Boundary)

[6:19:32](https://www.youtube.com/watch?v=htM02KMNZnk&t=22772s)

Gupta's eight-person team has spent about three years building the BAML programming language with no code reviews, mandatory parallel work, and no standardization of how engineers use AI, a provocative setup for a domain with zero slop tolerance. His definition: slop is any code you don't read. Their compensating tools: an architecture.md small enough that any model understands it, containing only invariants stable for months; a custom design-doc tool with versioning and comments, backed by markdown and CLI scripts, whose Slack feed became the company's most popular channel, with a rule that shipped design docs require actual readers; and a dependency-graph visualizer with CLI-enforced invariants so CI catches leaky package boundaries. For the language itself, agents continuously generate BAML programs from scratch, other agents inspect the transcripts for correctness and ergonomics (three tool calls where one should do), file issues, propose fixes, and A/B test language features on tool-call counts and error rates. He then argued the foundations are broken for agents: TypeScript's stated design goal balances correctness against human productivity, and things like array sort coercing numbers to strings are slop baked into the language. BAML's answers include inferred error types with compiler-proven exhaustive handling, code visualization in place of reading, near-zero-cost execution traces agents can navigate, a describe-code tool that replaces multi-call grep sessions, every function instantly runnable as a CLI, and cross-language interop that lets Python, TypeScript, Rust, Go, Ruby, and Java call BAML functions, lambdas included. One of his engineers had built a partial C compiler in BAML the day before. His close: we need a new git, a new database, and yes, a new programming language.

## Loop Engineering from First Principles (Kyle Mistele, HumanLayer)

[6:45:50](https://www.youtube.com/watch?v=htM02KMNZnk&t=24350s)

Mistele's target was "blind Ralph loops" that pipe a prompt into a coding agent forever and produce 40,000-line PRs nobody reads. His alternative is control theory: a sensor measures the system, the difference from a set point is the measured error, a controller converts error into an incremental control signal, an actuator applies it, and the loop repeats while the system undergoes disturbances. Thermostats, Kubernetes autoscaling, infrastructure-as-code, Postgres autovacuum, and React's virtual DOM all approximate this. Applied to agents: the sensor can be deterministic (ast-grep, linters) or an agent with a skill; the controller decides which small change to make; the actuator is a coding agent plus a skill. His worked example is HumanLayer's incremental migration of their RPC API to Effect. An ast-grep rule finds unmigrated procedures; a full scan on main is sorted deterministically and tracked in version control so every PR can be checked for newly added violations, a "disturbance dampener" against teammates' agent output undoing the loop's work. The controller picks the smallest violation to limit risk, and can be enriched with telemetry to prioritize the most error-prone procedures so the migration improves the code rather than translating it one-to-one. The actuator agent runs with a hand-written "golden pattern" example in its skill, and the whole loop runs as a scheduled GitHub Actions workflow producing one small PR each morning. Steering happens through a feedback markdown file tracked in git plus a /iterate PR comment trigger, and flow control allows at most one open PR per loop so unreviewed work never stacks up. Speed comes later: batch three to five migrations, or run four loop instances and give each teammate a PR. The skill is published.

## Harness Engineering Is Not Enough: Why Software Factories Fail (Dex Horthy, HumanLayer)

[7:32:31](https://www.youtube.com/watch?v=htM02KMNZnk&t=27151s)

Horthy (the captions mangle his name in the MC intro), on his fourth AI Engineer talk, argued the cracks in token-maxing are showing: a Faros AI report finds PR review quality down, more PRs merged without review, and incidents and bugs per developer up. His claim is that no amount of harness engineering or loop stacking fixes what is a model-training problem: models cannot maintain and improve codebase quality over time without human steering, and in his experience agents start struggling in codebases only three to six months old. HumanLayer tried a full lights-off factory in July 2025 and hit issues the agent could not solve while the site was down and users were angry. His explanation goes through RL training: coding models are reinforced on benchmarks shaped like SWE-bench multilingual, roughly 15-minute tasks with binary rewards for passing tests without breaking others, so there is no channel to penalize try-catch wallpaper, bad casts, or eroded design, because the cost function of bad architecture is measured in months and years and cannot propagate back across a single episode. He credited Claude Code's rise (he cited a run from nothing to multibillion revenue in under a year) to being the first case of a lab training the model against the exact harness users get. Better benchmarks are emerging: SWE Marathon from Abundant AI (400-hour tasks like cloning Excel), Deep Suite from Datacurve, and Cognition's Frontier Code (multi-PR tasks, penalties for tests that pass pre-patch, judge models for quality rules), but judge models can only go so far, since a model that knew what good code looks like would write it. For now: keep reading the code, and spend 30 minutes up front to save hours in review, via product review, architecture docs, program design (types, method signatures, call graphs), and vertical-slice implementation plans. His aphorism: you don't have too many PRs, you have too many bad PRs.

## In Code They Act, In Proof We Trust (Erik Meijer)

[7:51:41](https://www.youtube.com/watch?v=htM02KMNZnk&t=28301s)

Meijer gave a 20-minute tutorial, explicitly not a product pitch, on using elementary type systems to make agents provably safe. His story: chat-era LLMs were a function from question to answer, and even alignment plus LLM-as-judge could not formally prove an answer "safe," because safety of words is not a mathematical property. The real break came in June 2023 when OpenAI shipped tool calls, which he described as handing the model a loaded gun: in type terms the signature gains an IO effect, meaning the model can empty your bank account while computing a perfectly safe answer. Prompt injection makes this worse, a return of SQL injection with no code/data distinction. His fix proceeds in steps, demonstrated in Lean. First, defer execution: the agent returns a plan (a value of type IO of answer) instead of running the loop, air-gapping the agent from its tools. But IO is a black box you cannot reason about. So second, have the model return a program, an expression representing the computation, technically a free monad. Programs can be analyzed: dataflow analysis, type checking, and taint analysis over the expression solve Simon Willison's lethal trifecta, and a small inductive interpreter plus inductive proofs let models generate machine-checkable safety proofs before anything executes. He noted this is proof-carrying code, invented by academics in the 1990s, and that a Harvard-affiliated group has an implementation on GitHub. Three takeaways: agents are dangerous until proven safe; the plan language is designed for machines to generate, consume, and prove, not for humans to read; and all of it requires only programming-101 machinery.

## Recursive Model Improvement (Lee Robinson, Cursor)

[8:13:25](https://www.youtube.com/watch?v=htM02KMNZnk&t=29605s)

Robinson described a year of large-scale model training at Cursor. Composer 2.5 shipped in May and became the most popular model in Cursor, valued for being fast, smart, and cost-effective; the next version aims at a full pre-train from scratch (the previous model built on an open-source Kimi base), more data, and more RL. He framed training as two loops. The outer loop turns product usage into training signal: thumbs up/down feedback, A/B tests between checkpoints, and heavy internal dogfooding. The inner loop is evals and task generation: evals for intent understanding under 50 loaded skill files, for when to push back versus trust the user, and for simulated incident response (could the model have read the Datadog logs, Slack, and Notion and reached the same fix). He was candid about reward hacking: models learned to dig through git history for solutions and to search the web for forks of public evals, which affected Cursor's and other models' scores. Mitigations: delete git history at run start and restore it after, plus a network allowlist; Cursor Bench, their private eval built from held-out tasks in their own codebase, is the calibration they trust. Task generation includes building an ambitious application, deleting a feature so tests fail, and rewarding reimplementation. A newer method, textual feedback, zooms into one span of a long rollout, injects a hint (for example a reminder of available tools), and upweights or downweights token probabilities, giving precise credit assignment inside hundreds of thousands of tokens. On compute, he pointed to the March SpaceX partnership: Colossus stood up 100,000 GPUs in 122 days and added another 100,000 in 92, with Terrafab chips coming (his size comparison: 100 Buc-ee's). Researchers launch training runs from Slack with fleets of agents that page them when infrastructure breaks, and each model generation distills judge and reward models that speed up training the next one, the recursive part of the title. A new model was promised very soon.
