# Memory & Research Agents

Papers on what an agent should remember, how that memory should be structured, and how research-style agents actually behave when they run for hours or weeks. The through-line across all nine notes is that agent memory has quietly become a data-management problem (storage layout, retrieval operators, consolidation timing, maintenance cost, tenant isolation), and the honest answer these nine papers give to "which memory system wins?" is *it depends on the workload, and structure costs more than you think*. Several notes converge on the same uncomfortable finding: long context plus cheap RAG remains a strong default, graph memory earns its 30–150s-per-query cost only on genuinely relational cross-session work, and near-perfect recall benchmarks say almost nothing about whether an agent *acts* correctly on old information.

## How to read this area

Start with **Are We Ready For An Agent-Native Memory System?**, the only head-to-head benchmark of 12 real memory systems (Mem0, Zep, Cognee, MemOS, and nine more) across 11 datasets, and the note you want open before any vendor conversation. No architecture dominates, winners rotate by workload, and the graph systems' utility comes at a brutal 116–155s per query. Pair it with **Hierarchical Long-Term Semantic Memory for LinkedIn's AI Agents**, the production counterweight: a deployed KDD paper showing that shaping the memory tree around *business ownership* rather than semantic clusters beats HippoRAG/GraphRAG baselines while getting 0% cross-tenant leakage structurally. Third read: **Memory for Autonomous LLM Agents**, the survey that gives you the vocabulary (write–manage–read loop, five mechanism families) and the area's sharpest warning: models near-perfect on passive recall collapse to 40–60% when tasks require using memory across sessions.

The remaining six split by genre. Two are **architecture proposals with ablations**: GAM (consolidate to the graph at topic boundaries instead of every turn) and U-Mem (actively acquire memories via a cost-tiered teacher/tool/expert cascade; memory as a cheap substitute for fine-tuning). Two are **field studies of deployed behavior**: Agentic Search in the Wild (14.4M real search requests; agents hardcode retrieval depth and loop on near-duplicate queries) and Do Agents Need Semantic Metadata? (same model, different corpus, 65.7% precision swing). One is a **survey of the graph-memory design space** (taxonomy of six retrieval operator families; most products ship only one), and one is a **single-author deployment report** (Deep Researcher Agent: a 5K-character memory budget and zero-LLM-cost monitoring survived 30 days, challenging the premise that agent memory needs a database at all).

## Papers

| Paper | Core finding | Evidence strength |
|---|---|---|
| [Are We Ready For An Agent-Native Memory System?](are-we-ready-for-an-agent-native-memory-system.md) | No memory architecture dominates: winners rotate by workload (Zep tops LongMemEval, MemOS LoCoMo, MemoChat DB-Bench), and graph systems pay 116–155s/query vs 3.67s for lightweight ones; embedding RAG collapses (37.1→7.4 F1) as temporal distance grows. | Strong: 12 systems + 2 baselines, 11 datasets, module-level decomposition, public code; QA/conversation-heavy benchmarks though. |
| [Hierarchical Long-Term Semantic Memory for LinkedIn's AI Agents](hierarchical-long-term-semantic-memory-for-linkedin-s-ai-age.md) | A memory tree mirroring business ownership (project → recruiter → org) with multi-view nodes beats HippoRAG on correctness (0.892 vs 0.833) and gets 0.0% tenant leakage vs 30–92% in baselines; isolation comes from structure rather than filtering. | Strong: KDD 2026, deployed in LinkedIn's Hiring Assistant, 1,341 production queries + LongMemEval, nine baselines; one tree-shaped domain. |
| [Memory for Autonomous LLM Agents: Mechanisms, Architectures, and Evaluation](memory-for-autonomous-llm-agents-mechanisms-architectures.md) | High recall does not transfer to use: near-perfect LoCoMo systems drop to 40–60% on interdependent multi-session tasks; active-memory agents hit >80% vs ~45% for long-context-only; retrieval adds 200–500ms per call. | Moderate: useful synthesis and taxonomy, but single-author, not peer reviewed, numbers collated from cited papers. |
| [GAM: Hierarchical Graph-based Agentic Memory for LLM Agents](gam-hierarchical-graph-based-agentic-memory-for-llm-agents.md) | Decoupling encoding from consolidation (buffer locally, write to the topic graph only at detected semantic shifts) beats Mem0/MemoryOS/A-Mem (~13–22% F1) at the lowest token cost; the ablation shows the cheap event buffer matters most (−37.4% without it). | Solid: four backbones, clean ablations; gains shown on small open models and dialogue QA rather than tool-using agents. |
| [Towards Autonomous Memory Agents (U-Mem)](towards-autonomous-memory-agents.md) | Active, cost-aware memory acquisition (escalate failures through teacher → tools → expert) beats RL post-training (+5.8 pts over GRPO on HotpotQA at ~half the GPU hours), with only 23.2% of failures needing the expensive tier. | Solid: strong baselines and ablations; gains correlate r=0.888 with train/test overlap, and the "human expert" is another LLM. |
| [Agentic Search in the Wild](agentic-search-in-the-wild-intents-and-trajectory-character.md) | First at-scale field study of deployed search agents (14.44M requests): retrieval depth is hardcoded (91.6% of sessions), fact-seeking sessions degrade into near-duplicate loops (42.7% by step 9), and ~54% of productive new query terms come from previously retrieved evidence. | Strong for what it measures: 3.97M real sessions, released logs; no success labels, one API's traffic. |
| [Do Agents Need Semantic Metadata?](do-agents-need-semantic-metadata-a-comparative-study-in-age.md) | Same LLM, same framework, different corpus: routing to schema.org-annotated dataset search lifted FAIR-compliance precision 65.7% over open web search; the advantage is actionability, while topical relevance was a wash. | Solid but narrow: 58 English queries, one benchmark, single curated corpus (Google Dataset Search). |
| [Graph-based Agent Memory: Taxonomy, Techniques, and Applications](graph-based-agent-memory-taxonomy-techniques-and-applicat.md) | The graph-memory design space has six retrieval operator families and a four-stage life cycle (extract → store → retrieve → evolve); most production systems use only similarity search and skip evolution entirely. | Survey: organizes the space well (18 authors, companion repo) but reports no head-to-head numbers of its own. |
| [Deep Researcher Agent](deep-researcher-agent-an-autonomous-framework-for-24-7-deep.md) | A 30-day autonomous ML-experimentation agent ran on $0.08/day by never paying LLM tokens to watch training jobs (OS-level checks instead) and capping total memory at ~5K characters with lossy auto-compaction. | Weak-to-illustrative: single author, self-reported deployment, no named baselines; the cost patterns still generalize. |

## Builder takeaways

- **Benchmark cost-per-query alongside accuracy, and match structure to workload.** Graph memory's utility premium comes with a 30–40× latency/cost penalty; if your horizon fits the context window, long context + cheap RAG is still the default (Agent-Native Memory System).
- **Consolidate in batches rather than per turn.** Buffer interactions cheaply and write to structured memory at detected topic boundaries; write-through graph updates bake noise into long-term storage (GAM).
- **Shape memory around data ownership.** A hierarchy that mirrors who owns what gives you tenant isolation and clean deletion structurally; LinkedIn measured 0% leakage vs 30–92% for post-hoc filtering (HLTM).
- **Test action, not recall.** Evaluate whether the agent behaves correctly on information from ten sessions ago; recall-QA scores are where 40–60% collapses hide (Memory survey).
- **Add repetition detection and adaptive retrieval depth.** Deployed agents loop on near-duplicate queries and never vary K; both are cheap trajectory-level fixes (Agentic Search in the Wild).
- **Corpus beats model for data-seeking agents.** Route to structured/annotated sources first, open web for coverage fallback, and publish machine-readable metadata on your own data as agent UX (Semantic Metadata).
- **Don't pay tokens to wait.** Deterministic checks (process/GPU/log polls) while a long job runs, LLM calls only on state transitions; a hard character budget with auto-compaction can outlast a 30-day run (Deep Researcher).
- **Consider memory before fine-tuning.** A curated, utility-scored memory store matched RL post-training at half the compute; the self → teacher → tools → human escalation cascade is a reusable cost policy (U-Mem).

## Open questions

- When does structure pay for itself? The benchmark paper documents an orders-of-magnitude cost gap between graph and lightweight memory without consistent accuracy gains, but nobody has a decision rule for where the crossover sits on a given workload.
- Do the architecture papers' margins survive frontier models? GAM and U-Mem show gains mostly on small open backbones, and U-Mem's cascade requires a stronger teacher to exist, which fails exactly at the frontier.
- Almost everything here is dialogue/QA-benchmarked; long-horizon *tool-using* agent memory is only glancingly covered (two workloads in the benchmark paper, none in GAM), yet that's the deployment case builders actually face.
- Selective forgetting and contradiction handling remain the near-universal failure across systems; no paper in this folder ships a validated answer beyond expiry and human-visible audits.
- The field studies observe behavior without success labels: is a session that ends a session that succeeded? Until trajectory datasets carry correctness, "what deployed agents do" can't be separated from "what deployed agents get away with."
